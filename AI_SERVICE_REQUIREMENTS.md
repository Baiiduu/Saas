# AI 服务子需求规格说明（修订版）

> 本文档用于重定义本项目中的 AI 能力边界、数据来源、落库方式与工具调用规则。  
> 本次修订的目标不是“让 AI 更像聊天机器人”，而是把 AI 收口成一个**受权限约束、以工具取数和写库为准的企业内数据助手**。

---

## 1. 修订背景

当前 AI 模块存在以下方向性问题：

| 问题 | 说明 |
|------|------|
| AI 与前端数据不一致 | 用户在前端看到的任务、文档、审批，与 AI 回复中的结果不一致 |
| AI 与业务系统脱节 | AI 对业务对象的理解没有完全建立在正式 Tool / Service 之上 |
| 原始上下文污染 | 动态业务事实被直接塞进模型原始上下文、历史或记忆，导致“模型记住了旧数据” |
| 写操作语义错误 | AI 可能先在对话里声称“已创建 / 已修改”，但真实数据库不一定已完成写入 |
| 能力边界不清 | 哪些问题可以直接回答，哪些必须查 Tool，哪些必须确认后写库，没有被严格定义 |

本次修订后，AI 模块必须从“演示态对话能力”升级为：

**统一权限控制下的业务数据助手**

它的核心职责是：

1. 在当前用户权限范围内读取真实业务数据
2. 在需要时调用真实业务工具创建 / 修改数据并落库
3. 基于真实 Tool 结果回答，而不是基于过期上下文臆测
4. 对高风险写操作进行确认、审计与追踪

---

## 2. 模块重新定位

### 2.1 AI 助手应该做什么

AI 助手应当负责：

| 类别 | 允许能力 |
|------|----------|
| 通用理解 | 理解用户自然语言意图，识别这是“问数据”“写数据”“解释规则”“生成内容”中的哪一类 |
| 权限内取数 | 通过 Tool 查询任务、文档、审批、成员、里程碑等真实业务对象 |
| 结果整理 | 将 Tool 返回的结构化结果整理成自然语言、列表、摘要、引用 |
| 写操作代理 | 通过写工具创建任务、更新文档、发起审批等，并以数据库真实结果为准返回 |
| 安全控制 | 在权限不足、工具缺失、参数不足或高风险写操作时做拒绝、补充询问或确认 |

### 2.2 AI 助手不应该做什么

AI 助手不得：

| 禁止项 | 说明 |
|--------|------|
| 把动态任务列表 / 文档列表直接塞进原始系统提示词 | 动态业务事实必须通过 Tool 获取，不能作为原始上下文长期注入 |
| 把“上次查询到的结果”当成当前真实数据继续回答 | 当前态问题必须重新查 Tool |
| 在未落库前声称写操作已成功 | 创建、更新、审批等必须以真实业务写入结果为准 |
| 绕开正式业务服务直接拼装业务对象 | AI 和前端应共享同一后端业务服务和查询语义 |
| 在无工具支持时假装自己能查 | 缺少 Tool / Search 能力时必须明确说“当前系统不支持直接查询该信息” |
| 以 AI 独立身份超权读写 | AI 没有独立管理员身份，只能继承当前用户权限 |

## 3. 核心设计原则

| 原则 ID | 原则名称 | 说明 |
|---------|----------|------|
| P01 | Tool First | 只要用户问的是“当前系统里的真实业务数据”，AI 必须优先走 Tool |
| P02 | Same Source Of Truth | AI 与前端必须共用同一后端 Service / Query 语义，不能出现两套数据解释 |
| P03 | Write Through DB | 所有写操作必须经过正式业务写接口，成功后再回复用户 |
| P04 | Minimal Context | 原始上下文只保留身份、作用域、当前页面资源指针，不注入记忆、偏好或大段动态业务数据 |
| P05 | Permission Is Real | AI 的每次 Tool 调用都要经过真实权限校验，而不是仅靠前端展示或模型自觉 |
| P06 | Grounded Response | AI 的回答必须能追溯到 Tool 结果，并优先附带名称、ID、时间、引用等证据 |
| P07 | No Silent Fallback | Tool 不可用、权限不足、信息不足时，AI 应明确失败原因，不允许默默臆测 |

## 4. 典型业务问法与期望行为

| 用户问法 | 正确行为 | 错误行为 |
|----------|----------|----------|
| `有哪些委派给我的任务？` | 调用 `task.list`，附带“当前用户 assignee 过滤条件”，返回权限内真实任务列表 | 直接根据上下文、记忆或旧消息生成任务列表 |
| `文档中心有哪些与前端相关的文档？` | 调用文档查询 / 搜索工具，返回匹配文档名、创建者、更新时间、引用 | 直接从历史聊天内容中猜测文档名 |
| `小明上周是否提交了某个报告文档？` | 先解析“小明”对应成员，再查询指定时间范围内的文档 / 提交记录 / 审计记录 | 在没有查询结果的情况下凭上下文回答“应该提交了” |
| `帮我创建一个前端适配任务` | 调用 `task.create`，真实落库后返回任务 ID / 标题 / 所属团队 | 先在对话里说“已创建”，但数据库并没有记录 |
| `把这个文档改成正式版本` | 调用 `document.update_content` 或相关正式发布工具，按真实结果回复 | 把“要修改的内容”写进系统上下文，当成已经修改成功 |

## 6. 详细需求规格

### 6.1 AI01 - AI 助手定位与交互边界

| 属性 | 内容 |
|------|------|
| 需求标题 | AI 助手定位与交互边界 |
| 优先级 | 最高 |
| 功能描述 | AI 必须先判断用户请求类型，再决定是否调用 Tool / Skill。若用户是在问“系统当前数据”，则不能仅走普通模型回答。 |
| 前置条件 | 用户已登录；AI 会话已建立；存在当前租户与用户上下文。 |
| 触发条件 | 用户发送自然语言消息。 |
| 后置结果 | 请求被归类为：普通对话 / 读数据 / 写数据 / 生成内容 之一。 |
| 约束 | 普通聊天路径不得冒充真实查询路径；业务事实查询必须转入 Tool / Skill 流程。 |
| 验收要点 | 问“有哪些委派给我的任务”时，系统必须进入读数据路径，而不是纯模型直答。 |

### 6.2 AI02 - 权限内真实读

| 属性 | 内容 |
|------|------|
| 需求标题 | 权限内真实读 |
| 优先级 | 最高 |
| 功能描述 | 凡是涉及任务、文档、审批、成员、里程碑等当前业务事实的读取请求，AI 必须通过 MCP Tool 获取正式数据。 |
| 前置条件 | 已注册对应读工具；当前用户拥有相应读权限。 |
| 触发条件 | 用户询问当前任务、文档、审批、成员状态、统计、时间范围结果等事实性问题。 |
| 后置结果 | 返回结果来自 Tool 调用，而非会话记忆或原始上下文。 |
| 约束 | 若无适用 Tool，则必须明确告知当前系统不具备该查询能力，不允许猜测。 |
| 验收要点 | `有哪些委派给我的任务？` 必须触发 `task.list` 或等价读工具，并按当前用户过滤。 |

### 6.3 AI03 - AI 与前端查询一致性

| 属性 | 内容 |
|------|------|
| 需求标题 | AI 与前端查询一致性 |
| 优先级 | 最高 |
| 功能描述 | AI 读取类 Tool 必须复用前端所依赖的后端正式查询服务、过滤条件、排序规则与分页语义。 |
| 前置条件 | 前端存在对应业务页面；后端已形成共享查询 Service。 |
| 触发条件 | AI 调用任务列表、文档列表、审批列表、成员列表等查询工具。 |
| 后置结果 | 在相同用户、相同租户、相同团队、相同过滤参数下，AI 与前端看到的数据集合一致。 |
| 约束 | 不允许单独为 AI 写一套“近似查询逻辑”；如前端与 AI 能力不同，必须由同一后端能力配置裁剪。 |
| 验收要点 | 同一用户在任务页和 AI 查询“当前团队任务”时，结果集合、过滤逻辑与排序结果一致。 |

### 6.4 AI04 - AI 写操作真实落库

| 属性 | 内容 |
|------|------|
| 需求标题 | AI 写操作真实落库 |
| 优先级 | 最高 |
| 功能描述 | AI 创建任务、更新文档、发起审批、调整负责人等写操作，必须调用正式业务写 Tool，并以数据库真实写入结果作为回复依据。 |
| 前置条件 | 对应写 Tool 可用；用户具备写权限；必要参数完整。 |
| 触发条件 | 用户要求 AI 创建、修改、提交、更新某个业务对象。 |
| 后置结果 | 若写入成功，则返回真实对象 ID、标题、时间、状态；若写入失败，则返回明确失败原因。 |
| 约束 | 在数据库事务未成功前，AI 不得回复“已创建 / 已更新 / 已提交”。 |
| 验收要点 | `帮我创建一个前端适配任务` 后，数据库中必须存在该任务，AI 回复中应包含真实任务标识。 |

### 6.5 AI05 - 原始上下文最小化

| 属性 | 内容 |
|------|------|
| 需求标题 | 原始上下文最小化 |
| 优先级 | 最高 |
| 功能描述 | 原始系统上下文只允许包含身份、租户、团队、当前页面资源指针等最小运行信息；不引入记忆、偏好、会话摘要或任何跨会话延续内容。 |
| 前置条件 | AI 会话机制已存在。 |
| 触发条件 | 系统构建模型调用上下文。 |
| 后置结果 | 模型初始上下文保持极小化，不因旧会话、旧偏好或旧摘要产生污染。 |
| 约束 | 禁止把任务列表、文档列表、审批详情、大段文档正文、写操作结果、用户偏好、会话摘要写入原始系统上下文。 |
| 验收要点 | 新会话或既有会话进入模型调用时，原始上下文中不应出现历史意图、历史偏好、旧摘要或其他跨会话残留。 |

### 6.6 AI06 - 自然语言到 Tool / Skill 的路由

| 属性 | 内容 |
|------|------|
| 需求标题 | 自然语言到 Tool / Skill 的路由 |
| 优先级 | 高 |
| 功能描述 | 系统需具备意图判定与路由能力，将用户自然语言映射到合适的 Skill 或 Tool 组合。 |
| 前置条件 | 已存在 Tool Registry、Skill Registry、基础会话接口。 |
| 触发条件 | 用户发送自然语言消息，且该消息涉及系统真实数据或真实操作。 |
| 后置结果 | 请求被路由到：直接 Tool 执行、Skill 执行、补充询问、或普通聊天。 |
| 约束 | 当前态数据查询不得继续停留在纯模型路径。 |
| 验收要点 | 询问“有哪些委派给我的任务”时，系统应自动选择 `task.list` 路径，而不是只生成泛化建议。 |

### 6.7 AI07 - 检索类工具补齐

| 属性 | 内容 |
|------|------|
| 需求标题 | 检索类工具补齐 |
| 优先级 | 高 |
| 功能描述 | 为满足真实业务查询，需要补齐任务、文档、审批、成员的搜索、过滤、按人名解析、按时间范围查询等工具能力。 |
| 前置条件 | 基础读工具已存在。 |
| 触发条件 | 用户查询“与前端相关的文档”“小明上周提交的报告”“我负责的任务”等复杂筛选问题。 |
| 后置结果 | AI 具备正式的检索能力，而不是只能拿固定列表粗暴筛选。 |
| 约束 | 新增检索工具必须仍复用正式业务查询服务，不得引入 AI 私有数据查询通道。 |
| 验收要点 | 文档、任务等对象支持关键词、时间范围、创建者 / 负责人、状态等正式过滤条件。 |

### 6.8 AI08 - 有依据的回答与引用

| 属性 | 内容 |
|------|------|
| 需求标题 | 有依据的回答与引用 |
| 优先级 | 高 |
| 功能描述 | AI 在返回业务数据时，应附带对象名称、ID、时间、创建者、路径或引用摘要，使结果可回溯。 |
| 前置条件 | Tool 返回结构化结果。 |
| 触发条件 | 用户要求 AI 展示业务对象或判断某事实是否成立。 |
| 后置结果 | AI 回复不再是“口头概括”，而是可核对的结果说明。 |
| 约束 | 引用必须基于 Tool 结果，不得凭模型自造。 |
| 验收要点 | 文档查询结果至少应返回文档名、创建者、更新时间、必要时附引用片段或链接标识。 |

### 6.9 AI09 - 权限、确认与审计

| 属性 | 内容 |
|------|------|
| 需求标题 | 权限、确认与审计 |
| 优先级 | 高 |
| 功能描述 | 所有 Tool 调用必须经过真实权限校验；高风险写操作必须待确认；所有会话、Tool、Skill、权限拒绝和最终写入都必须审计。 |
| 前置条件 | 权限模块、AI 审计、确认流已存在。 |
| 触发条件 | 用户发起读写请求；系统执行 Tool / Skill。 |
| 后置结果 | 任意 AI 行为都可追踪、可解释、可回放。 |
| 约束 | AI 不得绕开确认直接执行高风险操作。 |
| 验收要点 | 查询行为可追溯；审批创建、任务指派等高风险动作有待确认状态和最终执行日志。 |

### 6.10 AI10 - AI 前端工作台收口

| 属性 | 内容 |
|------|------|
| 需求标题 | AI 前端工作台收口 |
| 优先级 | 高 |
| 功能描述 | AI 页面必须展示真实会话、Tool / Skill 执行过程、数据来源、待确认状态和错误信息，而不是只展示自然语言结果。 |
| 前置条件 | 后端已返回 Tool / Skill 步骤、状态与引用元信息。 |
| 触发条件 | 用户在 AI 工作台发送消息或触发 Skill。 |
| 后置结果 | 用户可区分“这是模型建议”“这是查到的真实数据”“这是已落库的结果”。 |
| 约束 | 不允许把 Tool 结果静默吞掉，只留下模型口头表述。 |
| 验收要点 | 用户能看到某次查询/创建到底调用了哪些 Tool、返回了哪些正式结果。 |

### 6.11 AI11 - AI 数据模型与服务边界

| 属性 | 内容 |
|------|------|
| 需求标题 | AI 数据模型与服务边界 |
| 优先级 | 高 |
| 功能描述 | 会话、消息、Tool 调用、Skill、审计必须独立建模；同时要拆分会话、路由、上下文、工具执行、引用整理、审计等服务边界。 |
| 前置条件 | 数据库迁移与后端模块拆分能力可用。 |
| 触发条件 | AI 模块进入正式工程化阶段。 |
| 后置结果 | AI 不再是一个大而散的聊天模块，而是边界清晰的企业内智能服务层。 |
| 约束 | 不允许把业务事实缓存逻辑、模型上下文逻辑、Tool 执行逻辑混在一个服务里。 |
| 验收要点 | 数据模型与服务职责清晰，便于后续测试、排错和迭代。 |

### 6.12 AI12 - AI 正确性验收

| 属性 | 内容 |
|------|------|
| 需求标题 | AI 正确性验收 |
| 优先级 | 高 |
| 功能描述 | AI 模块必须通过前后端一致性、权限边界、写库真实性、确认流、引用准确性等专项测试。 |
| 前置条件 | Tool / Skill 主链已接通。 |
| 触发条件 | AI 模块进入测试与验收阶段。 |
| 后置结果 | AI 能力不再以“看起来能聊”为准，而以“数据正确、权限正确、结果可追溯”为准。 |
| 约束 | 不以压力测试替代正确性测试；不以模型主观回答替代业务对账。 |
| 验收要点 | AI 查询到的任务/文档/审批结果应与前端同条件查询一致；写操作应与数据库最终结果一致。 |

## 7. 数据来源与落库规则

### 7.1 允许进入原始系统上下文的内容

| 类别 | 允许内容 |
|------|----------|
| 身份与作用域 | `tenantId`、`tenantName`、`userId`、`userName`、`role`、`teamId`、`teamName` |
| 页面指针 | 当前页面资源类型、当前页面资源 ID |
| 会话元数据 | `sessionId`、最近消息数量、当前操作模式 |

### 7.2 禁止进入原始系统上下文的内容

| 类别 | 禁止内容 |
|------|----------|
| 动态列表 | 任务列表、文档列表、审批列表、成员列表 |
| 动态统计 | 当前任务数、当前审批数、当前文档匹配数 |
| 大段资源正文 | 文档全文、审批详情全文、任务批量详情 |
| 未落库写结果 | “准备创建”的任务、“准备提交”的审批、“模型认为已经更新”的文档 |
| 易过期业务事实 | “当前由谁负责”“某文档是否存在”“本周提交了什么” |
| 跨会话延续 | 用户偏好、历史意图、会话摘要、任何记忆条目 |

## 10. 验收标准（修订版）

| 需求 ID | 验收条件 |
|---------|----------|
| AI01 | 用户自然语言请求会被正确分类；当前态业务查询不会继续走纯模型路径 |
| AI02 | 查询任务、文档、审批、成员等当前事实时，AI 必须走 Tool，并返回正式结果 |
| AI03 | 同用户、同团队、同过滤条件下，AI 与前端结果一致 |
| AI04 | AI 创建 / 修改业务对象后，数据库中必须存在一致结果；AI 回复应引用真实对象 ID 或标题 |
| AI05 | 原始上下文中不再保存动态任务 / 文档列表、历史偏好、会话摘要等易污染内容 |
| AI06 | 用户问“我的任务”“前端相关文档”“小明上周是否提交报告”等问题时，系统能正确路由到 Tool / Skill |
| AI07 | 检索类工具足以覆盖关键词、时间范围、成员解析等常见查询场景 |
| AI08 | AI 业务回答带有名称、时间、创建者、对象标识或引用摘要 |
| AI09 | 所有 Tool 调用均鉴权、可审计，高风险写操作需确认 |
| AI10 | 前端能展示真实 Tool / Skill 执行过程与数据来源 |
| AI11 | AI 数据模型与服务职责明确，便于追踪与扩展 |
| AI12 | AI 专项正确性测试通过：数据一致、权限正确、写库真实、上下文不污染 |

## 11. 结论

本项目中的 AI 不应再被定义为“带一点 Tool 的聊天接口”，而应被定义为：

**受权限控制、以正式业务 Tool 为唯一事实来源、以真实数据库写入为成功标准的企业内数据助手。**

后续所有 AI 开发与修复，应围绕以下主线推进：

1. **先定边界**：哪些必须查 Tool，哪些不能靠上下文
2. **再收口数据源**：AI 与前端必须共享同一后端事实来源
3. **再收口写路径**：AI 的创建和修改必须真实落库
4. **最后才谈体验增强**：如多模型、流式输出、提示词优化等
