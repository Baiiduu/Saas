# Requirements: SaaS多租户企业协作平台 — 完整需求分析

> 本文档基于《SaaS多租户企业协作平台需求规格说明书》（第17组课程大作业）整理生成，覆盖原型展示、V1落地与V2增强三个阶段。

---

## 1. 项目概述

### 1.1 项目背景

中小企业和项目型团队在日常协作中普遍面临以下问题：任务分散在聊天、电子表格和个人待办清单中，缺乏统一管理入口；文档与任务脱节，负责人不清晰；资源挂载与知识沉淀不足；成员间的通知反馈链路断裂。现有商业协作工具（如飞书项目、Teambition、Notion、Jira、Slack）功能强大但存在学习成本高、定制灵活性不足、多租户隔离能力有限等短板。

本项目拟构建一个**面向中小企业和项目型团队的多租户协作平台**，以"企业空间"为顶层组织单位、以"团队"为协作单元、以"任务、文档、仓库资源"为核心载体，集成通知、评论、附件、共享权限、审批流、协作资源图谱和大模型协作能力，将成员协作全过程系统化沉淀。

### 1.2 产品目标

| 序号 | 目标描述 |
|------|----------|
| G1 | 为企业提供一个统一的任务协作入口，替代分散在聊天、表格和个人待办中的执行过程 |
| G2 | 为团队提供结构化的任务、文档和仓库资源关联能力，减少信息孤岛 |
| G3 | 为课程作业提供清晰的分阶段开发路径，初次聚焦原型，后续逐步补齐完整功能 |
| G4 | 为后续大模型接入预留统一的后端能力边界，使模型可在授权范围内管理任务、文档与资源 |

### 1.3 核心设计理念

- **多租户优先**：一套平台实例支撑多个企业，账号、权限、数据、模型调用、审批和通知层面均严格隔离
- **协作闭环**：任务驱动执行 → 文档沉淀成果 → 资源关联上下文 → 通知串联反馈
- **可演进的架构**：功能按"原型可展示 → V1可落地 → V2可增强"分层推进
- **可验证交付**：每条需求均可通过页面原型、接口行为或测试用例验证

## 2. 用户角色分析

### 2.1 角色定义与权限矩阵

| 角色 | 标识 | 类型 | 权限范围 |
|------|------|------|----------|
| **企业 Owner** | `owner` | 企业内部 | 企业级全部管理权限，包含租户设置、计费（预留）、成员管理、审计日志查看 |
| **企业 Admin** | `admin` | 企业内部 | 除转让/注销企业外等同于 Owner，可管理团队、成员、审批模板、企业仪表盘 |
| **团队 Leader** | `leader` | 企业内部 | 团队级管理权限：管理团队成员、创建/编辑/归档团队、看板配置、审批流配置 |
| **普通成员** | `member` | 企业内部 | 可创建/编辑/评论/指派任务、上传文档、发起审批，属团队默认身份 |
| **只读成员** | `reader` | 企业内部 | 仅查看权限，可浏览团队任务、文档、看板，不可编辑或评论 |
| **外部访客** | `guest` | 企业外部 | 非企业内正式成员，通过授权链路访问指定资源（如单个文档或任务），可见范围由授权资源决定 |

### 2.2 角色关系说明

1. **Guest 为外部协作身份**，不属于企业内部基础角色；其可见范围完全由被授权资源决定
2. **文档级权限（打开/编辑/注释）**属于资源级控制，优先于普通成员的默认页面可见性
3. **审批模板的配置范围**不得超过当前操作者的企业或团队管理边界
4. 一个用户可以同时属于多个企业，在不同企业中可能扮演不同角色

### 2.3 各角色核心痛点

| 角色 | 核心痛点 |
|------|----------|
| Owner/Admin | 担心数据泄露、成员权限失控、租户隔离不彻底、审计追溯困难 |
| 团队 Leader | 任务进度不透明、资源关联混乱、审批流程阻塞、跨团队协作困难 |
| 普通成员 | 任务上下文缺失、文档找不到、通知过多或过少、资源引用不便 |
| 只读成员 | 需要了解项目进展但无编辑权限、信息获取不及时 |
| 外部访客 | 临时协作需频繁注册、权限申请流程冗长、访问链路不清晰 |
