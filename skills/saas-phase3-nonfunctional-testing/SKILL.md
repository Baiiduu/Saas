# SaaS 第三阶段非功能验证

## 核心目的

执行第三阶段测试，目标不是再证明功能“能不能用”，而是验证系统是否具备进入更真实使用环境的基础质量条件。

这一阶段要回答的问题包括：

- 系统在合理负载下是否表现正常
- 关键链路是否存在明显性能瓶颈
- 系统运行时是否容易失稳
- 日志、错误、审计、监控线索是否足够定位问题
- 安全基线是否存在明显缺口
- 当前构建、配置、部署准备是否达到可发布基础线

第一阶段解决的是功能正确性。  
第二阶段解决的是修复回归和边界稳健性。  
第三阶段解决的是系统在“接近真实使用”的条件下，是否具备上线前的基础非功能质量。

## 适用场景

在以下情况使用本 skill：

- 第一阶段和第二阶段已经完成，系统已有较稳定的功能基线
- 当前目标是验证非功能质量，而不是继续扩展功能
- 需要给出是否适合进入预发布或更广测试环境的依据
- 需要形成非功能测试报告和问题报告

不要在以下情况使用本 skill：

- 基础功能和权限还没站稳
- 目标是做极限压测或容量规划
- 目标是做正式渗透测试或红队演练
- 目标是做完整合规审计
- 目标只是修某个具体功能 Bug

## 你要扮演的角色

你在这一轮里是“非功能验证执行者”。

你的职责是：

- 根据系统现状建立可执行的非功能测试范围
- 选取关键路径做基础性能和稳定性验证
- 检查日志、错误、监控、审计等可观测性基础
- 检查关键安全基线是否存在明显缺口
- 检查环境、配置、构建、启动、迁移等发布准备度
- 形成结构化测试结论

除非用户明确要求，不要在这一阶段直接修复问题。

## 第三阶段与前两阶段的区别

第一阶段重点：

- 已实现功能是否正确
- 前后端是否对齐
- 权限和作用域是否基本成立

第二阶段重点：

- 修复是否有效
- 是否有回归
- 边界与异常路径是否可靠

第三阶段重点：

- 性能是否达到基础可用线
- 系统是否在持续运行和重复操作下保持稳定
- 问题发生时是否可定位
- 安全和配置是否存在低级但致命的缺口
- 当前系统是否具备进入下一环境的条件

## 测试边界

本阶段是“非功能验证”，不是“无限扩张测试”。

应重点覆盖：

- 基础性能验证
- 稳定性验证
- 可观测性验证
- 安全基线验证
- 发布准备验证

不要在本阶段扩张到：

- 极限高并发压测
- 真正的容量上限测算
- 深度攻防对抗
- 法务 / 合规 / 审计级别检查
- 长周期生产可用性统计

## 五类重点验证对象

### 1. 性能基础验证

验证系统在合理条件下是否有明显性能问题。

关注：

- 关键接口响应时间
- 关键页面首屏加载
- 列表加载、详情加载、保存提交、搜索筛选等关键操作
- 典型数据量下的响应表现

目标不是找极限，而是识别“明显不可接受”的性能问题。

### 2. 稳定性验证

验证系统在重复使用和持续运行下是否容易出错。

关注：

- 重复操作是否容易积累异常
- 长时间运行后服务是否明显退化
- 缓存、会话、任务、连接是否容易失稳
- 重试、重复提交、重复刷新是否引发异常状态

### 3. 可观测性验证

验证系统发生问题时，是否足够容易定位。

关注：

- 关键错误是否有日志
- 日志是否足够定位请求与上下文
- 是否存在请求标识、错误堆栈、用户上下文或作用域上下文
- 关键写操作是否有审计或操作记录
- 后端异常、前端异常是否能被有效捕获

### 4. 安全基线验证

本阶段不是渗透测试，而是安全基础线检查。

关注：

- 鉴权和鉴权失败行为是否合理
- 敏感信息是否被明文暴露在日志、接口或页面中
- 是否存在明显的未授权访问入口
- 配置项、密钥、调试开关是否处于危险状态
- 常见越权面是否仍残留明显缺口

### 5. 发布准备验证

验证系统是否具备基本的发布准备度。

关注：

- 环境变量是否完整
- 启动流程是否清晰
- 数据库迁移是否可执行
- 构建是否稳定
- 部署或初始化步骤是否可重复
- 关键依赖不可用时，系统是否有明确行为

## 非功能验证输入

在开始本阶段前，应尽量掌握：

- 第一阶段测试报告
- 第二阶段测试报告
- 当前剩余 Bug 列表
- 当前环境配置方式
- 关键架构或部署说明
- 日志、监控、审计相关设计信息

如果这些信息缺失，仍可执行测试，但必须在报告中说明非功能结论的置信度会下降。

## 数据准备原则

本阶段不建议使用空环境或极小样本环境直接得出性能和稳定性结论。

应准备：

- 真实感较强的测试数据
- 至少一组中等规模数据样本
- 能覆盖关键列表、筛选、详情、搜索、写入的样本
- 能覆盖典型权限和作用域的样本

目标不是“大数据”，而是“足以暴露明显非功能缺陷的数据量”。

如果环境资源有限，可以使用缩小版数据集，但必须在报告中说明限制。

## 推荐测试顺序

### 第一步：确认前置基线

- 确认第一、第二阶段是否已形成基本结论
- 确认当前环境可运行
- 确认系统至少可正常启动并跑通主链路

### 第二步：确定关键路径

选出最重要的非功能观察对象，例如：

- 登录
- 主导航进入
- 核心列表加载
- 核心详情页加载
- 关键创建 / 修改 / 提交流程
- 搜索 / 筛选 / 导出 / 上传等重操作

不要试图对所有路径做同等强度的非功能验证。

### 第三步：执行性能基础验证

检查：

- 接口响应时间
- 页面加载耗时
- 重操作是否显著卡顿
- 是否存在明显 N+1、重复请求、无分页、超大返回等问题

### 第四步：执行稳定性验证

检查：

- 重复提交
- 重复刷新
- 多次进入退出页面
- 连续批量操作
- 长时间运行后的资源泄漏或异常趋势

### 第五步：执行可观测性验证

检查：

- 请求是否可追踪
- 错误是否有记录
- 前后端问题是否能定位到入口
- 审计或操作记录是否足够支持排查

### 第六步：执行安全基线验证

检查：

- 未授权访问
- 失效 token 或错误身份访问
- 敏感字段泄露
- 调试信息暴露
- 危险默认配置

### 第七步：执行发布准备验证

检查：

- 构建是否稳定
- 启动是否清晰
- 配置是否完整
- 数据迁移是否可执行
- 依赖异常时是否有明确处理

### 第八步：整理结论

区分：

- 已达到基础可发布线
- 可运行但存在明显非功能缺口
- 当前不建议进入下一环境

## 测试方法要求

### 必须做的事

- 基于真实运行环境观察行为
- 记录测试前提和环境限制
- 将“现象”与“推测原因”分开写
- 关注关键路径而非平均路径
- 将非功能问题与功能问题分开归档

### 不要做的事

- 用一次偶然结果直接给出绝对结论
- 在低数据量环境下假装完成性能评估
- 把“没有报错”等同于“稳定”
- 把“能启动”等同于“具备发布准备度”
- 把“没有明显漏洞”等同于“安全”

## 缺陷分类建议

本阶段发现的问题，优先分成以下类别：

- 性能缺陷
- 稳定性缺陷
- 可观测性缺陷
- 安全基线缺陷
- 发布准备缺陷
- 环境限制或测试条件不足

每条问题尽量指出：

- 影响范围
- 触发条件
- 可复现程度
- 是否阻断下一阶段

## 文档产出要求

除非用户指定其他路径，默认在仓库根目录生成两个 Markdown 文件：

- `TEST_REPORT_PHASE3.md`
- `BUG_REPORT_PHASE3.md`

### 第三阶段测试报告必须包含

- 测试目标
- 非功能测试范围
- 当前环境说明
- 数据准备说明
- 关键路径清单
- 性能基础验证结论
- 稳定性验证结论
- 可观测性验证结论
- 安全基线验证结论
- 发布准备验证结论
- 环境限制与结论置信度
- 是否建议进入下一环境

### 第三阶段 Bug 报告必须包含

- Bug 编号
- 标题
- 严重级别
- 所属类别
- 影响模块或路径
- 前置条件
- 复现步骤
- 预期结果
- 实际结果
- 风险说明
- 建议修复方向

## 严重级别建议

- `critical`：阻断发布，或存在严重安全 / 稳定性 / 数据风险
- `high`：关键路径性能或稳定性明显不可接受，或可观测性严重缺失
- `medium`：存在明显优化和补强需求，但不必立刻阻断
- `low`：低风险非功能不足，建议后续治理

## 给测试智能体的标准提示词

将任务交给其他智能体时，可使用：

```text
Use $saas-phase3-nonfunctional-testing at /absolute/path/to/skills/saas-phase3-nonfunctional-testing to execute the third formal testing phase for a SaaS application.

Goal:
- validate baseline non-functional quality after functional correctness and regression stages
- focus on basic performance, stability, observability, security baseline, and release readiness
- use realistic test data and real execution behavior
- produce a phase-3 non-functional test report and a phase-3 bug report

Rules:
- do not do extreme load testing, capacity planning, deep penetration testing, or compliance audit work
- do not auto-fix during the testing pass
- explicitly record environment limits and confidence level of conclusions
- distinguish non-functional defects from functional defects
- state whether the system is ready for the next environment
```

## 可复用背景信息

如果需要给智能体补充背景，可以使用：

```text
This is the third formal testing phase for a SaaS application. Functional correctness and regression verification have already been established. The current goal is to validate baseline non-functional quality: performance, stability, observability, security baseline, and release readiness. Draw conclusions from real execution behavior, and keep the scope away from extreme load testing or formal offensive security work.
```

## 完成定义

一次测试运行只有在以下条件全部满足时才算完成：

- 已明确第三阶段非功能测试范围
- 已使用合理数据和环境执行验证
- 已覆盖关键路径的基础性能和稳定性观察
- 已完成可观测性、安全基线、发布准备检查
- 已记录环境限制与结论置信度
- 已产出第三阶段测试报告
- 已产出第三阶段 Bug 报告
- 已明确给出是否建议进入下一环境的结论
- 未在测试阶段混入修复动作，除非用户另行要求
