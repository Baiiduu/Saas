# 统一权限、共享权限与访客授权服务子需求规格说明

> 本文档作为《SaaS多租户企业协作平台软件需求规格说明书》中“角色权限、共享权限与访客授权”能力的细化子需求，用于把权限相关能力从分散规则收敛为统一服务，支撑后续业务模块、AI 模块、审计模块和测试工作的实施与验收。

---

## 1. 子模块定位

### 1.1 模块目标

统一权限、共享权限与访客授权服务用于为平台内所有资源访问和操作提供一致的鉴权基线。  
该服务不是单纯的角色枚举集合，而是统一管理以下能力：

- 企业级角色权限
- 团队级角色权限
- 资源级共享权限
- 访客授权
- 作用域继承与覆盖
- 接口级权限校验
- 页面级能力可见性
- 权限变更审计
- AI / 审批 / 文档等特殊场景下的授权边界

### 1.2 与总需求的关系

该子模块对应总需求中的以下条目：

| 需求 ID | 总需求描述 |
|---------|------------|
| F23 | 基于 RBAC 实现企业级角色权限控制 |
| F24 | 文档/任务级共享：设置打开、编辑、注释权限 |
| F25 | 访客授权：通过链接+访问码授权外部用户访问特定资源 |
| F26 | 权限继承与覆盖：资源级权限优先于角色级默认权限 |
| F27 | 审批模板配置范围不得超过操作者的企业/团队管理边界 |
| F84 | 租户间数据完全隔离 |
| F85 | 审计日志：记录关键操作（登录、创建/删除资源、权限变更等） |
| F91 | 模型调用权限控制：仅在授权范围内访问企业数据 |

### 1.3 当前实现基线

当前代码库已存在权限相关基础能力，但尚未形成统一权限服务：

| 项目 | 当前状态 |
|------|----------|
| 后端角色枚举 | 已存在 `OWNER / ADMIN / LEADER / MEMBER / READER / GUEST` |
| 后端权限矩阵 | `RbacService` 已定义资源-操作到最低角色的映射 |
| 后端角色守卫 | `RbacGuard` 已存在，但默认是非 strict 宽松模式 |
| 租户守卫 | `TenantGuard` 仅挂接 `X-Tenant-Id`，未做严格校验 |
| 资源级覆盖 | 当前仅有文档分享覆盖逻辑雏形 |
| 前端页面守卫 | 已有 `PermissionGuard`，但与后端并非所有场景严格对齐 |
| AI 权限 | MCP Tool 已声明 `requiredPermission`，但未统一收口执行 |
| 访客授权 | 需求层已有定义，正式闭环能力尚未完备 |

## 4. 详细需求规格

### 4.1 PS01 - 统一角色与权限模型

| 属性 | 内容 |
|------|------|
| 需求标题 | 统一角色与权限模型 |
| 版本规划 | 课程V1-收口版建立企业角色、团队角色、资源共享权限、访客授权四类能力的统一模型；课程V2-增强版扩展到更多资源和更多细粒度操作。 |
| 优先级 | 最高 |
| 功能描述 | 系统需要将企业角色、团队角色、资源级共享权限和访客授权统一到同一权限模型中。统一模型必须明确“谁可以对什么资源执行什么动作”，并定义资源级授权优先级、默认角色能力、访客受限能力和特殊场景边界。 |
| 前置条件 | 用户、企业、团队、资源等基础模型已存在；系统已定义基础角色。 |
| 触发条件 | 系统执行页面访问、接口访问、资源读取、资源修改、共享配置、访客访问、AI 工具调用等行为。 |
| 后置结果 | 权限规则具备统一判定标准；业务模块不再各自维护不一致的权限逻辑。 |
| 依赖关系 | 上游依赖：认证模块、租户模块、团队模块；下游影响：任务、文档、审批、AI、审计、前端权限展示。 |
| 主流程 | 1. 系统定义统一权限域；2. 定义角色层、共享层、访客层与作用域层；3. 业务模块调用统一权限服务进行判断；4. 权限服务返回允许、拒绝或需进一步校验的结果。 |

### 4.2 PS02 - 企业级与团队级角色管理

| 属性 | 内容 |
|------|------|
| 需求标题 | 企业级与团队级角色管理 |
| 版本规划 | 课程V1-收口版实现企业角色和团队角色的分配、变更、撤销和查询；课程V2-增强版可扩展更灵活的角色模板。 |
| 优先级 | 高 |
| 功能描述 | 系统支持企业内用户被赋予企业级角色，也可在团队内拥有团队级角色。角色变更必须由具备相应管理能力的操作者发起，并校验其变更边界。 |
| 前置条件 | 企业、团队和成员关系已存在；操作者已登录并具备成员管理权限。 |
| 触发条件 | Owner/Admin/Leader 在成员管理页面或后台接口中调整目标用户角色。 |
| 后置结果 | 目标用户在企业或团队内的有效角色更新成功，并影响后续页面可见性、接口访问和资源操作能力。 |
| 依赖关系 | 上游依赖：成员模块、团队模块；下游影响：全部业务模块、前端权限展示、审计模块。 |
| 主流程 | 1. 管理员进入成员与角色管理页；2. 选择目标成员；3. 选择企业角色或团队角色；4. 系统校验操作者是否具备变更目标角色的权限；5. 更新角色记录；6. 写入审计日志；7. 前端刷新有效权限。 |

### 4.3 PS03 - 统一接口级权限校验

| 属性 | 内容 |
|------|------|
| 需求标题 | 统一接口级权限校验 |
| 版本规划 | 课程V1-收口版实现后端统一鉴权执行链；后续增强可扩展到更多策略和资源级特例。 |
| 优先级 | 最高 |
| 功能描述 | 所有关键业务接口必须通过统一权限服务完成鉴权，不能仅依赖前端按钮显示、局部 controller 自行判断或页面跳转限制。统一鉴权至少校验身份、租户、团队、资源归属、操作类型和共享覆盖规则。 |
| 前置条件 | 用户已登录；请求携带租户上下文；目标资源存在或处于可校验状态。 |
| 触发条件 | 用户调用受保护接口，如创建、更新、删除、审批、共享、AI 调用等。 |
| 后置结果 | 合法请求被放行；越权请求被拒绝并返回结构化错误；权限拒绝被记录。 |
| 依赖关系 | 上游依赖：认证模块、租户模块、统一权限模型；下游影响：全部后端业务接口、AI 工具执行器。 |
| 主流程 | 1. 请求进入后端；2. 租户守卫解析并验证租户；3. 权限服务读取用户有效角色与共享授权；4. 权限服务判定目标操作是否允许；5. 若通过则执行业务逻辑；6. 若拒绝则返回错误并记录日志。 |

### 4.4 PS04 - 资源级共享权限

| 属性 | 内容 |
|------|------|
| 需求标题 | 资源级共享权限 |
| 版本规划 | 课程V1-收口版先覆盖文档与任务等核心资源；课程V2-增强版扩展到更多资源类型。 |
| 优先级 | 高 |
| 功能描述 | 系统支持对单个资源授予细粒度共享权限，如可打开、可编辑、可注释、可发起审批等。资源级共享权限优先于角色级默认页面可见性，并可作为精细授权手段。 |
| 前置条件 | 资源已创建；操作者具备资源管理或共享配置权限；目标用户为企业成员或符合共享条件的访问者。 |
| 触发条件 | 文档创建者、资源管理员或企业管理员在资源详情页设置共享权限。 |
| 后置结果 | 目标资源的共享权限被更新；有效访问者范围和可执行动作随之变化；共享变更被记录。 |
| 依赖关系 | 上游依赖：统一权限模型、资源模块；下游影响：文档预览、在线编辑、评论、审批、AI 权限范围。 |
| 主流程 | 1. 用户进入资源详情页；2. 选择共享对象或共享范围；3. 选择打开、编辑、注释等权限；4. 系统校验操作者是否具备共享权；5. 保存共享记录；6. 记录共享变更日志；7. 后续访问按共享规则执行。 |

### 4.5 PS05 - 访客授权服务

| 属性 | 内容 |
|------|------|
| 需求标题 | 访客授权服务 |
| 版本规划 | 课程V2-增强版实现链接 + 访问码 + 失效时间的受控访客授权；后续可扩展更细粒度的访客行为限制。 |
| 优先级 | 中 |
| 功能描述 | 系统支持对外部访客授予指定资源的受限访问能力。访客不属于企业内部成员，其访问范围必须严格受授权资源、访问码、失效时间和允许动作控制。访客不能因为单个链接而扩展到企业内其他资源。 |
| 前置条件 | 目标资源支持访客共享；操作者具备访客授权权限；访客访问链路已启用。 |
| 触发条件 | 管理员或资源所有者在资源授权页创建访客访问链接。 |
| 后置结果 | 生成访客授权记录；访客可在授权有效期内访问指定资源；越界访问被拒绝。 |
| 依赖关系 | 上游依赖：资源级共享权限、统一权限模型；下游影响：文档外链、对外演示场景、审计模块。 |
| 主流程 | 1. 用户进入资源授权页；2. 选择“创建访客访问”；3. 设置访问码、过期时间和允许操作；4. 系统生成授权链接；5. 访客访问链接并输入访问码；6. 系统校验授权有效性；7. 若通过则进入受限资源页。 |

### 4.6 PS06 - 权限继承、覆盖与作用域规则

| 属性 | 内容 |
|------|------|
| 需求标题 | 权限继承、覆盖与作用域规则 |
| 版本规划 | 课程V1-收口版明确租户、团队、资源三层规则和冲突解决顺序；课程V2-增强版扩展到访客和 AI 特殊场景。 |
| 优先级 | 最高 |
| 功能描述 | 系统必须明确定义权限继承、覆盖和作用域规则。企业角色提供默认能力，团队角色提供团队内能力，资源级共享权限可对具体资源做覆盖。所有判断必须同时考虑租户边界、团队边界和资源边界。 |
| 前置条件 | 企业、团队、资源和角色模型已存在；统一权限服务已建立。 |
| 触发条件 | 同一用户在不同团队、不同资源、不同页面或不同接口下发起操作。 |
| 后置结果 | 权限冲突可被一致解决；不会因角色、团队和资源规则并存而出现判断漂移。 |
| 依赖关系 | 上游依赖：统一权限模型；下游影响：文档共享、AI 权限、审批模板配置、访客授权。 |
| 主流程 | 1. 系统读取租户上下文；2. 读取用户企业角色和团队角色；3. 读取资源级共享或访客授权；4. 按预定义优先级执行判定；5. 输出最终允许或拒绝结果。 |

### 4.7 PS07 - 审批模板与 AI 工具的边界授权

| 属性 | 内容 |
|------|------|
| 需求标题 | 审批模板与 AI 工具的边界授权 |
| 版本规划 | 课程V1-收口版实现审批模板配置边界校验；课程V2-增强版实现 AI 工具权限与统一授权服务的联动。 |
| 优先级 | 高 |
| 功能描述 | 审批模板的配置、审批人范围、可发起人范围不得超出操作者的企业或团队管理边界。AI 工具调用也必须复用统一权限服务，确保模型不能读取或修改超出当前用户授权范围的数据。 |
| 前置条件 | 审批模块和 AI 模块已接入统一权限服务；操作者已登录。 |
| 触发条件 | 用户配置审批模板、发起审批，或通过 AI 调用 MCP Tool / Skill 访问业务资源。 |
| 后置结果 | 审批模板配置和 AI 工具调用都被限制在真实管理边界内；越界行为被拒绝并记日志。 |
| 依赖关系 | 上游依赖：统一接口级权限校验、审批模块、AI 模块；下游影响：企业治理可信度、AI 可控性。 |
| 主流程 | 1. 用户配置审批模板或触发 AI 工具；2. 系统识别目标团队、目标资源和所需动作；3. 权限服务校验配置或调用边界；4. 若通过则继续执行；5. 若失败则返回拒绝结果并写日志。 |

### 4.8 PS08 - 权限变更审计与拒绝日志

| 属性 | 内容 |
|------|------|
| 需求标题 | 权限变更审计与拒绝日志 |
| 版本规划 | 课程V1-收口版实现角色变更、共享变更、越权拒绝日志；课程V2-增强版扩展到访客授权和 AI 权限事件。 |

| 前置条件 | 审计模块可用；权限服务已统一接入。 |

| 主流程 | 1. 用户发起权限相关操作；2. 系统完成鉴权与变更；3. 系统写入审计日志；4. 若为拒绝则记录拒绝原因与上下文；5. 管理员后续可查询记录。 |

### 4.9 PS09 - 前后端权限对齐

| 属性 | 内容 |
|------|------|
| 需求标题 | 前后端权限对齐 |
| 版本规划 | 课程V1-收口版实现前端 `PermissionGuard` 与后端真实权限矩阵对齐；课程V2-增强版扩展到更多细粒度按钮和 AI 工具可见性。 |
| 优先级 | 高 |
| 功能描述 | 前端页面级守卫、按钮显示逻辑、操作入口可见性必须与后端真实鉴权规则保持一致。前端可用于提升体验，但不作为最终安全边界。任何前端显示为无权限的操作，后端仍必须拒绝；任何前端显示为有权限的操作，也必须能被后端真实放行。 |
| 前置条件 | 前端已具备权限 Hook/Guard；后端权限矩阵已稳定。 |
| 触发条件 | 用户进入页面、查看按钮、提交表单、发起接口调用。 |
| 后置结果 | 前端交互与后端鉴权不再长期漂移；权限相关 Bug 更容易测试和定位。 |
| 依赖关系 | 上游依赖：统一权限模型、前端权限 Hook/Guard；下游影响：页面体验、测试阶段正确性验证。 |
| 主流程 | 1. 前端加载当前用户有效权限；2. 页面根据权限显示入口；3. 用户发起操作；4. 后端使用同一套权限语义进行校验；5. 若前后端结果不一致，则记录为权限契约缺陷。 |

### 5.2 权限判定优先级

| 顺序 | 规则 |
|------|------|
| 1 | 必须先校验身份是否合法、会话是否有效 |
| 2 | 必须校验租户上下文是否存在且有效 |
| 3 | 必须校验目标资源是否属于当前租户 |
| 4 | 先判断企业 / 团队角色默认权限 |
| 5 | 再判断资源级共享是否对目标资源形成覆盖 |
| 6 | 若为访客访问，则只允许走访客授权规则，不继承企业内部默认权限 |
| 7 | 若为 AI / 审批等特殊能力，则在基础权限之上再校验场景边界 |

## 6. 验收标准

| 需求 ID | 验收条件 |
|---------|----------|
| PS01 | 系统存在统一权限服务，可对角色权限、共享权限、访客授权和特殊场景权限进行一致判定 |
| PS02 | 企业角色和团队角色可以被分配、变更和撤销，且不允许低权限用户越级分配更高权限 |
| PS03 | 关键业务接口必须经过统一权限校验；仅隐藏前端按钮而后端放行的情况视为不通过 |
| PS04 | 对文档、任务等资源的共享权限可生效，且资源级共享可覆盖默认角色权限 |
| PS05 | 访客只能访问被授权的指定资源，不能越界访问企业内部其他内容 |
| PS06 | 同一用户在不同租户、不同团队、不同资源下的权限冲突可按统一优先级得到一致结果 |
| PS07 | 审批模板配置和 AI 工具调用都不能超出操作者的真实企业/团队/资源边界 |
| PS08 | 角色调整、共享变更、访客授权和越权拒绝均可审计和追溯 |
| PS09 | 前端页面守卫和后端真实鉴权保持一致，权限正确性测试可验证通过 |

## 7. 实施注意事项

| 项目 | 说明 |
|------|------|
| TenantGuard 收口 | 当前 `TenantGuard` 仅读取头部未验证租户合法性，统一权限服务落地前必须收口租户校验 |
| RbacGuard 模式 | 当前 `RbacGuard` 默认非 strict，需避免“守卫放行，业务层忘记校验”的风险 |
| 资源级共享扩展 | 当前资源级覆盖主要体现在文档分享逻辑，后续需抽象成统一共享策略 |
| AI 联动 | AI 工具权限必须复用同一权限服务，避免另起一套授权判断 |
| 前后端契约 | 前端 `PermissionGuard`、权限 Hook 与后端矩阵应建立统一语义映射，避免长期漂移 |

## 8. 结论

统一权限、共享权限与访客授权服务在本项目中应被定义为“横切全部资源访问与操作的基础治理层”，而不是零散散落在业务模块中的局部判断。  
后续开发、测试和验收，应以本文件中的 PS01-PS09 需求表为基线推进。
