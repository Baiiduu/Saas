# 需求规格写作

## 概述

本技能用于把“方向、能力、想法、功能块”写成真正可实施、可拆任务、可测试、可验收的需求规格。
目标不是写产品宣传稿，也不是写泛泛的设计说明，而是产出结构清晰、字段完整、颗粒度稳定的规格文档。

## 适用任务

在以下情况使用本技能：

- 用户要写“子需求文档”
- 用户要把一个服务、模块或能力细化成正式需求
- 用户要按已有 SRS / 规格说明书风格补写新章节
- 用户要把原本泛泛的需求说明改成表格式规格
- 用户要产出需求总表、单条需求规格表、验收标准表
- 用户提供了 Word / Markdown / PDF 规格文档，希望仿照其中的写法扩写新需求
- 用户希望需求文档后续可直接用于开发、测试、验收和任务拆分

不要在以下情况使用本技能：

- 用户只是要头脑风暴点子
- 用户只是问“这个功能值不值得做”
- 用户只是要简短 PRD 摘要
- 用户只是要接口文档，而不是需求规格
- 用户只是要实现方案，不需要需求视角

## 工作目标

使用本技能时，你要输出的内容应满足：

- 能被开发拿去拆任务
- 能被测试拿去写用例
- 能被评审拿去做范围确认
- 能和已有规格书结构对齐
- 能落到具体模块、具体能力、具体边界
- 不停留在抽象口号层

## 基本原则

### 1. 先对齐“规格风格”，再写内容

如果项目里已经存在正式需求规格说明书、SRS、Word 规格文档、Markdown 需求文档或固定表格格式：

- 先抽样查看它是怎么写单条需求的
- 识别它的表格字段
- 识别它的术语、版本命名、优先级表达、阶段划分
- 用同样风格继续写

不要自创一套和现有文档风格完全不同的格式。

### 2. 先确认“写什么”，再确认“怎么写”

在落文档前，必须先明确：

- 这是一个模块、服务，还是某条能力链路
- 它属于哪个版本阶段
- 它和已有需求的关系是什么
- 当前代码是否已有基线实现

### 3. 规格文档必须具体，不要泛泛而谈

避免这类空泛表达：

- “系统应支持智能化能力”
- “要加强权限控制”
- “要优化用户体验”

应改写成：

- 谁可以做什么
- 在什么前提下触发
- 系统应返回什么结果
- 会影响哪些上下游模块
- 通过什么流程完成
- 测试时如何验证

### 4. 默认输出“需求总表 + 单条需求规格表”

如果用户没有特别指定格式，优先使用：

1. 文档定位与版本规划
2. 需求总表
3. 单条需求规格表
4. 补充约束
5. 验收标准
6. 实施注意事项

## 标准工作流

### 第一步：识别规格来源

优先判断是否存在上游规格基线：

- Word 需求规格说明书
- Markdown 需求文档
- 已有子需求文档
- 项目代码中的真实实现
- 产品原型或接口行为

如果存在正式规格来源：

- 必须先抽样几条已有需求
- 提炼其表格字段和语言风格

如果不存在正式规格来源：

- 使用本技能的默认规格模板

### 第二步：确认需求边界

在写之前先收敛边界：

- 该文档是写“一个服务”还是“一个模块”
- 是否聚焦一个主题
- 是否属于已有总需求中的某几个需求点
- 是否需要与当前代码基线对齐

必须避免把多个无关主题揉进一份子需求文档。

### 第三步：建立需求总表

先输出一张总表，把该子模块拆成若干需求项。

总表至少包含：

| ID | 需求描述 | 阶段 |
| -- | -------- | ---- |

要求：

- ID 要稳定、可引用
- 描述要一句话说清楚能力边界
- 阶段要和主规格书一致

### 第四步：展开单条需求规格

单条需求默认采用如下字段：

| 属性     | 内容                                       |
| -------- | ------------------------------------------ |
| 需求标题 | 该需求项名称                               |
| 版本规划 | 该需求在哪个阶段落地，基础版与增强版怎么切 |
| 优先级   | 高 / 中 / 低，或沿用原规格书写法           |
| 功能描述 | 这条需求到底要解决什么问题、提供什么能力   |
| 前置条件 | 执行这条需求前必须满足什么状态             |
| 触发条件 | 用户或系统在什么情况下触发这条需求         |
| 后置结果 | 需求执行完成后系统达到什么结果             |
| 依赖关系 | 它依赖哪些上游模块，又影响哪些下游模块     |
| 主流程   | 用 1,2,3... 描述主路径                     |

如有必要，再补：

- 异常流程
- 权限约束
- 数据约束
- 审计要求
- 非功能要求

### 第五步：补齐约束和验收

当文档进入“可交付”状态时，应补：

- 能力矩阵
- 范围边界
- 验收标准
- 风险与实施注意事项

不要停在“有需求表但不能验收”的状态。

## 推荐输出结构

如果用户没有指定结构，默认用下面这套：

### A. 文档头部

- 文档名称
- 文档定位
- 与上游需求的映射关系
- 当前实现基线
- 版本规划

### B. 需求总表

用于先给读者建立全局地图。

### C. 详细需求规格

逐条展开，采用“属性 | 内容”表格。

### D. 补充约束

适合放：

- 能力矩阵
- 权限矩阵
- 工具清单
- 数据域范围
- 角色边界

### E. 验收标准

适合按“需求 ID | 验收条件”表输出。

### F. 实施注意事项

适合记录：

- 当前代码基线的限制
- 迁移或收口难点
- 与其他模块的强依赖

## 写作要求

### 要做到

- 用正式规格语言写，不写宣传话术
- 尽量让每条需求都能被测试
- 尽量让每条需求都有明确边界
- 让需求 ID、阶段、优先级保持一致
- 当已有代码存在时，写清“当前基线”
- 当已有上游规格存在时，显式对齐其表格结构

### 不要这样写

- 不要把设计方案直接冒充需求
- 不要把实现细节写得比需求本身还重
- 不要只写愿景，不写触发、结果和流程
- 不要把多个能力混成一条需求
- 不要在没有核对上游文档风格时自由发挥

## 验收标准

一份由本技能产出的需求文档，至少应满足：

- 有清晰的文档定位
- 有需求总表
- 有可引用的需求 ID
- 有逐条规格表
- 有版本规划和优先级
- 有前置条件、触发条件、后置结果和依赖关系
- 有验收标准
- 和上游规格书风格一致，或至少内部一致

## 可直接复用的提示词

```text
Use $requirements-spec-writing to write a formal requirements specification or sub-requirements document.

Goal:
- turn high-level requirements, module ideas, service capabilities, or existing implementation analysis into a structured specification document
- avoid generic prose
- align with the style of any existing SRS, Word spec, Markdown requirements doc, or current project requirement tables when available
- produce a requirements summary table and detailed per-requirement specification tables

Rules:
- write in formal specification style, not product marketing language
- if an existing requirements document exists, sample its field structure first and follow it
- each requirement should be concrete, bounded, testable, and traceable
- include version planning, priority, preconditions, triggers, postconditions, dependencies, and main flow
```

## 完成定义

当且仅当满足以下条件时，本技能的一次使用才算完成：

- 已识别并对齐上游规格风格或已明确采用默认模板
- 已收敛需求边界
- 已形成需求总表
- 已形成逐条规格表
- 已补充验收标准
- 文档可直接用于开发、测试或评审
