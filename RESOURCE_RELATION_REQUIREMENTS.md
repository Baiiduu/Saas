# 资源关系与画布编排服务子需求规格说明

> 本文档作为《SaaS多租户企业协作平台软件需求规格说明书》中“资源关系、协作拓扑与可视化编排”能力的细化子需求。  
> 本子需求的目标不是继续增强当前的“只读关系图页面”，而是把资源关系正式升级为一个**可落节点、可建关系、可分层组织、可与真实业务资源联动的画布式资源编排服务**。

---

## 1. 子模块定位

### 1.1 模块目标

资源关系与画布编排服务用于把团队中的核心协作对象抽象为节点，并允许用户在画布上组织、关联、查看和维护这些节点之间的关系。  
该服务统一承载以下能力：

- 资源节点抽象
- 画布拖放与布局
- 节点父子层级
- 节点间关系边
- 里程碑为父节点的结构化编排
- 员工、任务、文档、Git 仓库等异构节点共存
- 节点点击跳转与详情联动
- 图数据持久化与导出

### 1.2 与总需求的关系

该子模块对应总需求中的以下条目：

| 需求 ID | 总需求描述 |
|---------|------------|
| F40 | 资源关系图：展示文档、任务、成员、里程碑之间的协作关系 |
| F41 | 资源节点可视化：支持不同类型资源的节点样式区分 |
| F42 | 支持导出资源关系图 |
| F84 | 租户间数据完全隔离 |
| F85 | 审计日志：记录关键操作（创建/删除资源、关系变更等） |

### 1.3 当前实现基线

当前代码库已有“资源关系图”基础能力，但与目标子需求差距较大：

| 项目 | 当前状态 |
|------|----------|
| 前端画布组件 | 已使用 `reactflow` 做基础只读展示 |
| 前端交互能力 | 仅支持浏览、过滤、点击节点、导出占位图片 |
| 后端图接口 | 仅支持以 `task/document` 为中心查询已有边 |
| 后端图模型 | 仅有 `tenant.graph_edges`，没有画布、节点布局、分组、节点元数据模型 |
| 支持资源类型 | 实际只稳定支持 `task`、`document`，前端枚举中有 `member/milestone` 但未形成真实闭环 |
| 编辑能力 | 不支持把资源拖上画布、不支持建边、不支持调整父子关系 |
| Git 仓库节点 | 尚未支持 |

## 2. 版本规划

| 版本 | 目标 | 范围说明 |
|------|------|----------|
| V1 基础版 | 建立画布、节点、关系边、里程碑父子节点和 Git 仓库节点的正式模型 | 包含节点落画布、关系编辑、基本跳转、持久化 |
| V2 增强版 | 增强自动布局、模板画布、批量导入、关系校验和 AI / Git 深度联动 | 包含布局算法、关系推荐、仓库元数据同步、跨模块自动建边 |

## 4. 详细需求规格

### 4.1 RR01 - 统一节点模型

| 属性 | 内容 |
|------|------|
| 需求标题 | 统一节点模型 |
| 版本规划 | V1 基础版先覆盖里程碑、任务、文档、成员、Git 仓库五类节点；V2 增强版可扩展审批、资源库、AI Agent 等节点类型。 |
| 优先级 | 最高 |
| 功能描述 | 系统必须为画布中的所有对象建立统一节点模型。每个节点至少具备节点 ID、节点类型、显示名称、关联业务资源 ID、归属团队、位置信息、展示样式和可选扩展元数据。 |
| 前置条件 | 已存在基础业务对象模型；团队上下文已确定。 |
| 触发条件 | 用户创建画布、拖入资源、查看画布或读取关系图数据。 |
| 后置结果 | 画布中的不同对象可按统一结构被渲染、保存、查询和编辑。 |
| 依赖关系 | 上游依赖：任务、文档、里程碑、成员模块；下游影响：画布、关系边、导出、AI。 |
| 主流程 | 1. 系统定义节点类型枚举；2. 为各类型资源定义映射规则；3. 画布保存节点基础字段与扩展字段；4. 前端按节点类型渲染。 |

### 4.2 RR02 - 画布与视图管理

| 属性 | 内容 |
|------|------|
| 需求标题 | 画布与视图管理 |
| 版本规划 | V1 基础版支持团队级单画布和基础保存加载；V2 增强版支持多画布、命名视图、模板画布。 |
| 优先级 | 高 |
| 功能描述 | 系统支持在团队范围内创建资源关系画布，保存当前节点与边的布局，后续可再次打开并继续编辑。 |
| 前置条件 | 用户已进入团队，且拥有资源关系查看或编辑权限。 |
| 触发条件 | 用户进入资源关系页，首次创建画布，或加载既有画布。 |
| 后置结果 | 画布配置被持久化，团队成员在权限范围内可复用同一画布视图。 |
| 依赖关系 | 上游依赖：团队模块、权限模块；下游影响：节点管理、关系边、导出。 |
| 主流程 | 1. 用户打开资源关系页；2. 系统加载团队默认画布；3. 若不存在则允许创建新画布；4. 用户编辑后保存；5. 系统持久化画布结构。 |

### 4.3 RR03 - 节点拖放与布局

| 属性 | 内容 |
|------|------|
| 需求标题 | 节点拖放与布局 |
| 版本规划 | V1 基础版支持从资源池拖放节点到画布并保存坐标；V2 增强版支持自动吸附、对齐线、自动布局。 |
| 优先级 | 最高 |
| 功能描述 | 系统应提供资源池或资源面板，允许用户把任务、文档、成员、里程碑、Git 仓库节点拖到画布中，并调整其位置。节点位置信息必须可保存。 |
| 前置条件 | 画布已创建；资源池可加载团队内真实资源。 |
| 触发条件 | 用户在资源池中选择资源并拖入画布，或在画布中移动节点。 |
| 后置结果 | 节点出现在画布中，且位置被保存；下次进入仍能保持布局。 |
| 依赖关系 | 上游依赖：统一节点模型、画布管理；下游影响：父子编排、关系边、导出。 |
| 主流程 | 1. 系统加载资源池；2. 用户拖动资源到画布；3. 系统为其创建画布节点记录；4. 用户拖动调整坐标；5. 系统保存新位置。 |

### 4.4 RR04 - 里程碑父子编排

| 属性 | 内容 |
|------|------|
| 需求标题 | 里程碑父子编排 |
| 版本规划 | V1 基础版支持里程碑作为父节点或容器节点挂载子节点；V2 增强版支持折叠、批量挂载和进度汇总。 |
| 优先级 | 最高 |
| 功能描述 | 系统支持把里程碑作为大节点或容器节点，用于承载任务、文档、成员、Git 仓库等子节点。子节点既属于画布，也可属于某个里程碑节点的局部结构。 |
| 前置条件 | 团队中已存在里程碑或允许新建里程碑节点。 |
| 触发条件 | 用户创建里程碑节点，或将其他节点挂到里程碑节点下。 |
| 后置结果 | 里程碑与子节点关系被保存，前端可按层级展示。 |
| 依赖关系 | 上游依赖：里程碑模块、统一节点模型；下游影响：关系边、布局、统计汇总。 |
| 主流程 | 1. 用户创建或选中里程碑节点；2. 用户拖入任务/文档/成员/Git 仓库节点；3. 系统建立父子关系；4. 画布按层级渲染；5. 保存结构。 |

### 4.5 RR05 - 关系边管理

| 属性 | 内容 |
|------|------|
| 需求标题 | 关系边管理 |
| 版本规划 | V1 基础版支持手工建立、编辑和删除关系边；V2 增强版支持关系规则校验、样式配置和自动推荐。 |
| 优先级 | 最高 |
| 功能描述 | 用户可在任意两个节点之间建立关系边，并为边指定关系类型，如“依赖”“产出”“负责人”“参考”“归属”“关联仓库”等。 |
| 前置条件 | 画布中已存在两个可连接节点；用户具备编辑权限。 |
| 触发条件 | 用户在画布上拖拽连线，或通过节点操作菜单创建关系。 |
| 后置结果 | 关系边被保存并可视化展示；后续可编辑标签、删除关系或查看关系详情。 |
| 依赖关系 | 上游依赖：统一节点模型、画布编辑；下游影响：导出、自动关系、AI 可读性。 |
| 主流程 | 1. 用户选择源节点和目标节点；2. 指定关系类型；3. 系统校验该关系是否允许；4. 保存关系边；5. 前端刷新显示。 |

### 4.6 RR06 - Git 仓库节点接入

| 属性 | 内容 |
|------|------|
| 需求标题 | Git 仓库节点接入 |
| 版本规划 | V1 基础版支持手工录入 Git 仓库 URL 和显示名称；V2 增强版支持读取仓库平台元数据和分支信息。 |
| 优先级 | 高 |
| 功能描述 | 系统支持把 Git 仓库作为一种正式节点类型加入画布。创建 Git 仓库节点时，用户需填写仓库 URL，可选填写名称、描述、默认分支等。节点点击后可跳转到外部仓库地址。 |
| 前置条件 | 用户具备画布编辑权限；输入的仓库 URL 符合基本格式。 |
| 触发条件 | 用户在资源池中新建 Git 仓库节点，或把已有仓库配置项拖入画布。 |
| 后置结果 | Git 仓库节点出现在画布中，并可与任务、文档、里程碑、成员建立关系。 |
| 依赖关系 | 上游依赖：统一节点模型、画布管理；下游影响：前端跳转、资源整合、V2 仓库联动。 |
| 主流程 | 1. 用户选择“新增 Git 仓库节点”；2. 输入仓库 URL 和名称；3. 系统校验格式；4. 保存节点；5. 允许用户建立关系边；6. 点击节点可外跳仓库。 |

### 4.7 RR07 - 节点点击与业务联动

| 属性 | 内容 |
|------|------|
| 需求标题 | 节点点击与业务联动 |
| 版本规划 | V1 基础版支持跳转到任务、文档、成员或仓库；V2 增强版支持右侧详情面板、快捷操作和预览。 |
| 优先级 | 高 |
| 功能描述 | 节点不是纯展示对象。用户点击节点后，应能够进入对应业务对象的详情页，或在仓库节点上跳转到仓库 URL。 |
| 前置条件 | 节点已正确绑定业务资源 ID 或外部 URL。 |
| 触发条件 | 用户点击画布上的节点。 |
| 后置结果 | 系统打开对应业务详情、侧边详情面板或外部仓库。 |
| 依赖关系 | 上游依赖：统一节点模型、路由系统；下游影响：资源详情、协作导航效率。 |
| 主流程 | 1. 用户点击节点；2. 系统识别节点类型；3. 根据节点配置跳转到任务/文档/成员/里程碑详情或外部仓库；4. 若资源已失效则提示。 |

### 4.8 RR08 - 权限与作用域控制

| 属性 | 内容 |
|------|------|
| 需求标题 | 权限与作用域控制 |
| 版本规划 | V1 基础版实现团队级查看/编辑权限和资源引用边界控制；V2 增强版扩展共享画布、访客视图和更细粒度编辑权限。 |
| 优先级 | 最高 |
| 功能描述 | 资源关系画布必须受真实权限控制。用户只能把自己权限范围内的任务、文档、成员、里程碑、Git 仓库节点加入画布，也只能查看和编辑其有权访问的画布。 |
| 前置条件 | 权限服务可用；团队、资源和用户关系已可判定。 |
| 触发条件 | 用户加载画布、拖入节点、建边、删除节点或跳转节点。 |
| 后置结果 | 未授权资源不会出现在可拖拽资源池中；越权节点操作被拒绝；跨租户和跨团队引用被阻止。 |
| 依赖关系 | 上游依赖：统一权限服务、团队模块、资源模块；下游影响：AI、审计、测试。 |
| 主流程 | 1. 用户进入资源关系页；2. 系统解析团队上下文与用户权限；3. 加载权限内画布和资源池；4. 编辑时再次进行真实鉴权；5. 越权操作返回拒绝。 |

### 4.9 RR09 - 导出、审计与历史

| 属性 | 内容 |
|------|------|
| 需求标题 | 导出、审计与历史 |
| 版本规划 | V1 基础版支持 JSON / 图片导出和关键变更审计；V2 增强版支持版本历史、回滚与差异比较。 |
| 优先级 | 中 |
| 功能描述 | 系统应支持导出当前画布，并记录节点创建、节点移位、关系创建、关系删除、父子结构变更等关键事件。 |
| 前置条件 | 画布已存在内容；审计模块可用。 |
| 触发条件 | 用户导出画布，或对节点、边、父子关系做变更。 |
| 后置结果 | 当前图谱可供导出和复盘；关键编辑行为具备审计追踪能力。 |
| 依赖关系 | 上游依赖：画布管理、审计模块；下游影响：运维排查、测试验收、协作复盘。 |
| 主流程 | 1. 用户发起导出；2. 系统按当前视图导出；3. 用户编辑节点或边；4. 系统记录审计日志；5. 后续支持按条件查询。 |

### 4.10 RR10 - 自动关系与增强能力

| 属性 | 内容 |
|------|------|
| 需求标题 | 自动关系与增强能力 |
| 版本规划 | V2 增强版 |
| 优先级 | 中 |
| 功能描述 | 系统可基于业务事实或规则自动生成部分关系，如“任务产出某文档”“成员负责某任务”“任务归属某里程碑”“仓库支撑某任务”等。自动关系应与手工关系共存，并可区分来源。 |
| 前置条件 | V1 节点、边、权限和审计能力已稳定。 |
| 触发条件 | 用户开启自动关系生成，或系统检测到明确业务映射。 |
| 后置结果 | 画布可自动补全部分关系，降低纯手工连边成本。 |
| 依赖关系 | 上游依赖：任务、文档、成员、里程碑、仓库节点模型；下游影响：AI、分析、自动布局。 |
| 主流程 | 1. 系统扫描业务事实；2. 识别候选关系；3. 标记关系来源为“系统建议”或“自动生成”；4. 用户确认或系统直接落边；5. 后续可编辑。 |

## 5. 补充约束

### 5.1 节点类型矩阵（V1）

| 节点类型 | 是否映射真实资源 | 是否可做父节点 | 是否可建关系边 | 是否支持点击跳转 |
|----------|------------------|----------------|----------------|------------------|
| 里程碑 | 是 | 是 | 是 | 是 |
| 任务 | 是 | 否 | 是 | 是 |
| 文档 | 是 | 否 | 是 | 是 |
| 成员 | 是 | 否 | 是 | 是 |
| Git 仓库 | 否（外部资源） | 否 | 是 | 是 |

### 5.2 首批关系类型建议

| 关系类型 | 含义 | 典型连接 |
|----------|------|----------|
| depends_on | 依赖 | 任务 -> 任务 |
| produces | 产出 | 任务 -> 文档 |
| owned_by | 负责 | 任务 -> 成员 |
| belongs_to | 归属 | 任务/文档/成员/Git 仓库 -> 里程碑 |
| references | 引用 | 文档 -> 文档 / 文档 -> 任务 |
| backed_by_repo | 仓库支撑 | 任务/文档 -> Git 仓库 |

### 5.3 资源池范围约束

- 资源池只展示当前团队、当前租户、当前用户权限范围内的资源。
- 未授权资源不得通过手工输入 ID 被拖入画布。
- 外部 Git 仓库节点虽然不属于内部业务资源，但其创建、可见和编辑仍受团队权限控制。

### 5.4 非功能边界

- 本子需求当前不要求做知识图谱推理。
- 本子需求当前不要求做大型图数据库迁移。
- V1 以团队级中小规模画布可用为目标，不以超大图谱性能优化为首要目标。

## 6. 验收标准

| 需求 ID | 验收条件 |
|---------|----------|
| RR01 | 画布节点至少支持里程碑、任务、文档、成员、Git 仓库五类，且有统一结构 |
| RR02 | 团队可创建并重复打开同一资源关系画布，节点和边可被保存 |
| RR03 | 用户可从资源池把真实任务、文档、成员等节点拖到画布，并保持位置 |
| RR04 | 里程碑节点可容纳子节点，父子结构可保存和再次加载 |
| RR05 | 用户可在两个节点之间建立关系边，并在重开画布后保留 |
| RR06 | 用户可新建 Git 仓库节点，填写 URL 后能在节点点击时跳转 |
| RR07 | 点击任务/文档/成员/里程碑节点能进入对应详情或正确联动 |
| RR08 | 用户无法把无权限资源加入画布，也无法查看无权限画布 |
| RR09 | 节点、关系变更具备审计记录，画布支持至少一种正式导出 |
| RR10 | V2 中系统可基于业务事实生成至少一类自动关系 |

## 7. 实施注意事项

### 7.1 与当前实现的关键差异

- 当前后端仅有 `GraphEdge`，不够支撑画布、节点布局、父子结构和 Git 节点。
- 当前前端 `ResourceGraphPage` 是“资源详情上的只读关系视图”，而不是“团队级画布编排页”。
- 当前 `GraphService` 只支持以 `task/document` 为中心查询，不支持加载整张团队画布。

### 7.2 推荐的数据模型方向

V1 建议至少新增以下正式模型：

- `GraphCanvas`
- `GraphNode`
- `GraphNodeParent`
- `GraphEdge`（可保留并增强）
- `GraphExternalResource` 或 `GraphRepoNode`

### 7.3 推荐实施顺序

1. 先确定节点、画布、边的数据模型
2. 再做团队级画布查询与保存接口
3. 再做前端资源池 + 拖拽落点 + 位置保存
4. 再做里程碑父子结构
5. 再接 Git 仓库节点
6. 最后做导出、审计和自动关系

### 7.4 与其他子需求的关系

- 与权限子需求强依赖：资源池可见性、画布编辑权、节点跳转权都受统一权限服务约束
- 与 AI 子需求有关联：后续 AI 可以读取或生成资源关系，但不能绕开正式图谱模型
- 与任务、文档、里程碑模块强依赖：这些模块是首批正式节点来源
