# SaaS 第二阶段回归与边界测试

## 核心目的

执行第二阶段测试，目标不是再次做“是否能跑通”的基础确认，而是验证：

- 第一阶段发现并修复的问题是否真的修好
- 修复是否引入新的回归
- 关键链路在边界输入和异常路径下是否仍然正确
- 权限和作用域控制在更复杂场景下是否仍然可靠
- 系统是否接近“可持续迭代”的稳定状态

第一阶段回答的是：

当前功能是否基本正确。

第二阶段回答的是：

在修复和迭代之后，系统是否仍然稳定，且不会轻易被边界条件击穿。

## 适用场景

在以下情况使用本 skill：

- 第一阶段正确性测试已完成
- 已有一批 Bug 被修复或部分修复
- 需要验证修复有效性，而不是仅看代码修改
- 需要执行面向回归与边界的第二轮正式测试
- 需要产出第二阶段测试报告和剩余 Bug 报告

不要在以下情况使用本 skill：

- 系统还没完成第一阶段基础正确性验证
- 当前目标是性能、压测、稳定性压榨、容量评估
- 当前目标是安全攻防、渗透、模糊测试
- 当前目标是需求探索或产品验收，而非回归验证

## 你要扮演的角色

你在这一轮里是“回归与边界测试执行者”。

你的职责不是重新发明测试范围，而是基于已有结论进一步验证：

- 修复是否有效
- 修复是否外溢破坏其他功能
- 异常路径是否被妥善处理
- 权限、状态、作用域的边界是否清晰

除非用户明确要求，不要在测试阶段直接修复问题。

## 第二阶段与第一阶段的区别

第一阶段的重点是：

- 已实现功能的基础正确性
- 前后端是否能对上
- 基本权限是否工作
- 页面和接口是否能正常跑通

第二阶段的重点是：

- 已修问题的回归验证
- 关键链路的交叉影响
- 边界输入与异常输入
- 状态流转完整性
- 权限绕过风险的复验
- 数据一致性与重复操作正确性

## 测试边界

本阶段只测试以下内容：

1. 第一阶段覆盖过的关键功能
2. 第一阶段失败过且已声明修复的缺陷
3. 修复影响到的关联模块
4. 当前代码已实现、且具备边界验证意义的能力

不要把本阶段扩张为“全面重新测一遍所有东西”。

本阶段应该以风险优先方式聚焦：

- 曾出过问题的地方
- 高频主链路
- 跨模块关联点
- 状态机、权限和数据写入场景

## 重点关注对象

第二阶段优先验证以下六类问题：

### 1. 缺陷修复验证

对第一阶段每个已修 Bug，验证：

- 是否真正修复
- 是否只修表象未修根因
- 是否只在单一路径修复，其他入口仍旧存在

### 2. 回归风险验证

验证修复是否影响：

- 原有可用页面
- 原有接口结构
- 原有正常权限行为
- 原有成功写入流程

### 3. 边界输入验证

验证系统在边界输入下的行为，例如：

- 空值
- 超长值
- 临界值
- 非法枚举值
- 缺字段
- 重复提交
- 已删除 / 已失效 / 已关闭对象的后续操作

### 4. 状态流转验证

如果系统存在业务状态或生命周期，验证：

- 合法状态迁移能成功
- 非法状态迁移被拒绝
- 回退路径是否合理
- 终态对象是否仍可被错误修改

### 5. 权限稳健性验证

第二阶段权限验证应比第一阶段更强，重点看：

- 是否存在前端限制但后端未限制的情况
- 是否存在跨角色、跨团队、跨作用域的残留绕过
- 是否存在读可拒绝但写未拒绝，或反过来的不一致
- 是否存在部分接口修了权限，关联接口没修

### 6. 数据一致性验证

验证多次操作后系统数据是否仍一致：

- 修改后再次读取是否正确
- 删除后是否仍可被引用
- 批量更新后列表与详情是否一致
- 关联对象更新后引用是否同步
- 重复操作是否幂等或至少行为明确

## 种子数据原则

第二阶段通常不应从零开始随机造一套新数据，而应：

1. 尽量复用第一阶段数据
2. 在其基础上补充“边界型样本”
3. 为修复场景准备最小复现数据

补充数据应优先覆盖：

- 第一阶段失败过的场景
- 需要验证的修复路径
- 状态边界
- 权限边界
- 作用域边界

如果第一阶段的数据已经不可用或不完整，可以重建，但必须记录：

- 与第一阶段相比，哪些数据被复用
- 哪些是新注入的边界数据
- 哪些数据专门用于验证修复

## 最低数据覆盖建议

在不绑定具体业务模块的前提下，至少补充：

- 第一阶段暴露问题所需的最小复现样本
- 至少 1 组异常输入或边界输入样本
- 至少 1 组越权或跨作用域验证样本
- 至少 1 组状态流转验证样本
- 至少 1 组重复提交 / 重复修改 / 重复删除样本

如果系统包含批量操作、关联对象、层级对象、导入导出或流程节点，这些都应优先纳入边界验证。

## 测试顺序

推荐按下面顺序执行：

### 第一步：读取第一阶段结论

优先读取并整理：

- 第一阶段测试报告
- 第一阶段 Bug 报告
- 已修复问题列表
- 用户补充说明的修复范围

如果没有这些输入，就先向用户说明第二阶段测试会缺少回归基线。

### 第二步：建立回归测试清单

至少列出三类对象：

- 必须复验的已修问题
- 必须复测的关键主链路
- 必须补测的边界和异常场景

### 第三步：准备修复验证数据

- 复用第一阶段样本
- 补充边界样本
- 为每个重点缺陷准备最小复现条件

### 第四步：执行修复验证

逐条检查已声明修复的问题：

- 是否修好
- 是否仍可复现
- 是否只在部分入口修好

### 第五步：执行回归测试

围绕被修改模块及其关联模块，验证：

- 正常路径是否仍可用
- 数据结构是否变了
- 旧接口或旧页面是否被破坏

### 第六步：执行边界与异常测试

重点验证：

- 空输入
- 非法输入
- 重复输入
- 非法状态迁移
- 对无权限对象的访问或修改
- 对失效对象的二次操作

### 第七步：整理剩余风险

区分：

- 已修复并通过验证
- 已修复但仍有残留问题
- 未修复
- 新回归问题
- 待确认问题

## 测试方法要求

### 必须做的事

- 以真实运行行为为准
- 对已修问题做“复现前提 -> 验证修复”闭环检查
- 至少验证一轮正常路径和一轮异常路径
- 对权限问题做反向验证，不只验证成功场景
- 对修复外溢模块做最小回归检查

### 不要做的事

- 看到代码改了就默认修复成功
- 只验 happy path 就宣布回归通过
- 把未复现当成已修复，而不检查复现条件是否成立
- 把环境问题、数据问题和代码问题混在一起
- 没有第一阶段基线却伪装成标准回归测试

## 缺陷分类建议

第二阶段发现的问题，优先分成以下类别：

- 修复无效
- 部分修复
- 回归缺陷
- 边界条件缺陷
- 状态流转缺陷
- 权限缺陷
- 作用域隔离缺陷
- 数据一致性缺陷
- 环境 / 数据准备问题

每条问题都尽量标记：

- 是否属于第一阶段遗留
- 是否属于本轮新回归
- 是否属于新发现的边界问题

## 文档产出要求

除非用户指定其他路径，默认在仓库根目录生成两个 Markdown 文件：

- `TEST_REPORT_PHASE2.md`
- `BUG_REPORT_PHASE2.md`

### 第二阶段测试报告必须包含

- 测试目标
- 与第一阶段的关系
- 回归基线来源
- 测试范围
- 修复验证范围
- 补充的边界场景
- 环境信息
- 数据准备说明
- 已验证通过项
- 回归失败项
- 新发现问题
- 权限与作用域复验摘要
- 当前剩余风险

### 第二阶段 Bug 报告必须包含

- Bug 编号
- 标题
- 严重级别
- 所属模块
- 问题类别
- 是否为回归
- 是否为第一阶段遗留
- 前置条件
- 复现步骤
- 预期结果
- 实际结果
- 初步归因
- 建议修复方向

## 严重级别建议

- `critical`：修复后仍存在核心阻塞，或出现越权、越域、严重数据错误
- `high`：关键链路回归、关键状态错误、关键写操作不正确
- `medium`：边界处理错误、部分回归、局部逻辑错误
- `low`：低风险不一致、提示不准、非关键边界问题

## 给测试智能体的标准提示词

将任务交给其他智能体时，可使用：

```text
Use $saas-phase2-regression-testing at /absolute/path/to/skills/saas-phase2-regression-testing to execute the second formal test phase for a SaaS application.

Goal:
- verify that previously fixed issues are actually fixed
- detect regressions caused by recent changes
- test boundary inputs, invalid operations, status transitions, permission robustness, and scope isolation
- produce a phase-2 test report and a phase-2 bug report

Rules:
- do not do performance, stress, load, benchmark, or broad security testing
- do not auto-fix during the testing pass
- use the phase-1 report and bug list as regression baseline when available
- verify real running behavior, not just code changes
- explicitly distinguish fixed issues, unfixed issues, regressions, and new boundary findings
```

## 可复用背景信息

如果需要给智能体补充背景，可以使用：

```text
This is the second formal testing phase for a SaaS application. The phase-1 correctness pass has already established the basic functionality baseline. The current goal is to verify fixes, detect regressions, and validate boundary behavior, permission robustness, scope isolation, and state-transition integrity. Confirm behavior through actual system execution rather than code intent alone.
```

## 完成定义

一次测试运行只有在以下条件全部满足时才算完成：

- 已建立第二阶段回归基线
- 已确认需要复验的修复项
- 已执行关键链路回归验证
- 已执行边界与异常验证
- 已执行权限与作用域复验
- 已区分修复成功、残留问题与新回归
- 已产出第二阶段测试报告
- 已产出第二阶段 Bug 报告
- 未在测试阶段混入修复动作，除非用户另行要求
