# SaaS 第一阶段正确性测试

## 核心目的

执行一次“正确性优先”的正式测试阶段，验证系统当前已经实现的功能是否：

- 能正常访问
- 能正确返回
- 能正确渲染
- 能正确写入
- 能正确拒绝越权操作
- 能在合理的数据前提下稳定完成基础业务流程

该阶段不是为了评估系统性能，也不是为了做未来功能验收，而是为了回答一个更基础的问题：

当前已经做出来的东西，是否真的正确。

## 适用场景

在以下情况使用本 skill：

- 一个 SaaS 应用准备进入第一次系统性测试
- 前后端都已有一定实现，需要验证真实对接情况
- 需要验证权限、作用域、接口返回、页面行为是否一致
- 需要先产出测试结论和 Bug 清单，再决定是否进入修复阶段

不要在以下情况使用本 skill：

- 目标是做压测、性能测试、容量测试、稳定性压榨
- 目标是做渗透、安全攻防、模糊输入攻击测试
- 目标是测试尚未实现的未来需求
- 目标是边测边改，不保留测试阶段的独立结论

## 你要扮演的角色

你不是功能开发者，也不是产品许愿代理。

你在这一轮里是“正确性测试执行者”，职责是：

- 建立可复现的测试前提
- 枚举当前可测范围
- 注入合理数据
- 验证真实系统行为
- 记录失败与偏差
- 输出可交付文档

除非用户单独要求，不要在测试阶段直接修复问题。

## 测试边界

本 skill 只测试“已经确认实现”的能力。

判断某项能力是否纳入测试，必须同时满足：

1. 代码中已经存在实现
2. 存在可访问入口
3. 当前环境中具备可验证条件

入口可以是：

- 前端页面
- 后端接口
- 系统内部工作流
- 后台管理入口

不要把“未来要做”“文档里写了但代码没落地”“界面有占位但链路没接完”的内容记为功能 Bug。
它们可以记为“未实现”或“超出本阶段范围”，但不是本轮测试缺陷。

## 重点关注对象

本阶段优先验证以下五类正确性：

### 1. 契约正确性

检查前端与后端之间是否真正对齐：

- 请求参数是否匹配
- DTO / schema / shared types 是否一致
- 响应字段是否满足前端实际依赖
- 枚举值、可空字段、分页结构、错误结构是否兼容

### 2. 接口正确性

检查后端接口是否正确：

- 正常输入返回正确数据
- 非法输入返回合理错误
- 数据写入后结果可被再次读取验证
- 返回结构稳定，不靠隐式猜测

### 3. 页面正确性

检查前端页面是否正确：

- 页面是否能打开
- 是否有白屏、致命渲染异常、循环加载
- 列表、详情、表单、提交结果是否符合真实数据
- 成功态和失败态是否可理解

### 4. 权限正确性

检查系统是否正确允许和拒绝操作：

- 应有权限的人能成功
- 不该有权限的人被拒绝
- 前端隐藏不等于后端真正拒绝
- 写操作必须验证服务端权限，而不是只看按钮状态

### 5. 作用域正确性

检查数据访问边界是否正确：

- 用户是否只看到自己该看到的数据
- 团队 / 工作空间 / 组织 / 租户之间是否隔离
- 同级主体之间是否越界可读或可写
- 缺失上下文时是否有明确定义行为

## 种子数据原则

开始测试前，必须先构造“合理且最小化”的测试数据。

种子数据应满足：

- 能覆盖核心业务流程
- 能覆盖主要权限层级
- 能覆盖作用域边界
- 能覆盖读写行为
- 可重复构建
- 可明确说明来源

优先使用以下方式注入数据：

1. 真实业务流程
2. 管理接口或内部工具
3. 数据库脚本、ORM 脚本、种子脚本

如果必须直接写库，可以写，但必须在报告中记录：

- 注入方式
- 注入内容
- 这样做的原因
- 对测试结果的影响

## 最低数据覆盖建议

在不绑定具体业务模块的前提下，至少准备：

- 至少 1 个顶层作用域主体
  - 例如租户、组织、工作空间、站点
- 如果系统存在二级作用域，再至少准备 2 个二级主体
  - 例如团队、项目、空间、部门
- 至少 4 到 6 个不同权限层级的用户
- 至少 1 组核心业务对象样本
- 如果业务对象有状态流转，覆盖多个关键状态
- 如果系统有附件、文件、文档或富文本能力，覆盖至少 1 组真实样本
- 如果系统有流程、审批、发布、工单、订单、结算等链路，覆盖至少 1 条完整最小链路

不要为了“看起来全”而注入大量脏数据。
优先保证数据代表性，而不是数量。

## 测试顺序

推荐按下面顺序执行：

### 第一步：确认测试环境

- 依赖是否可安装
- 前后端是否能启动
- 数据库和依赖服务是否可用
- 当前构建是否至少可运行

### 第二步：确认可测范围

- 列出当前真正能测的功能
- 排除尚未实现或明显未接通的功能
- 建立本轮测试清单

### 第三步：准备测试数据

- 构造最小但有效的数据集
- 确保覆盖权限和作用域
- 确保核心链路可走通

### 第四步：执行接口与契约测试

- 验证请求结构
- 验证响应结构
- 验证错误结构
- 验证分页与元信息
- 验证前端对返回值的真实依赖

### 第五步：执行前端功能测试

- 页面访问
- 列表加载
- 详情展示
- 表单提交
- 基本创建 / 修改 / 删除 / 状态变更

### 第六步：执行权限与作用域测试

- 允许场景
- 拒绝场景
- 越权场景
- 错误上下文场景

### 第七步：整理结论

- 测试通过项
- 已确认缺陷
- 阻塞项
- 开放问题
- 残余风险

## 测试方法要求

### 必须做的事

- 以真实运行的系统行为为准
- 用最小路径复现问题
- 把失败结果和预期结果明确分开
- 同时验证前端表现和后端真实行为
- 对权限问题做正反两组验证

### 不要做的事

- 只读代码不跑系统就下结论
- 只看前端按钮是否可见就判断权限正确
- 只测 happy path
- 只记录现象，不追踪最小复现条件
- 在测试文档中混入未经确认的猜测

## 缺陷分类建议

测试中发现的问题，优先分成以下类别：

- 前端缺陷
- 后端缺陷
- 前后端契约不一致
- 权限缺陷
- 作用域隔离缺陷
- 环境 / 数据准备问题
- 已知限制或尚未实现项

每条缺陷都应尽量指出：

- 表现在哪里
- 怎么复现
- 预期是什么
- 实际发生了什么
- 更像是哪一层的问题

## 文档产出要求

除非用户指定其他路径，默认在仓库根目录生成两个 Markdown 文件：

- `TEST_REPORT_PHASE1.md`
- `BUG_REPORT_PHASE1.md`

### 测试报告必须包含

- 测试目标
- 测试范围
- 测试环境
- 种子数据说明
- 测试方法
- 已覆盖功能
- 通过项
- 失败项
- 阻塞项
- 权限验证摘要
- 契约对齐摘要
- 当前残余风险

### Bug 报告必须包含

- Bug 编号
- 标题
- 严重级别
- 所属模块
- 影响入口
- 前置条件
- 复现步骤
- 预期结果
- 实际结果
- 初步归因
- 建议修复方向

## 严重级别建议

- `critical`：核心流程不可用，或出现越权、越域、错误主体访问
- `high`：重要已实现功能不可用，或读写结果明显错误
- `medium`：功能部分错误，但存在可绕行路径
- `low`：低风险不一致、展示问题、非关键错误提示问题

## 给测试智能体的标准提示词

将任务交给其他智能体时，可使用：

```text
Use $saas-correctness-testing at /absolute/path/to/skills/saas-correctness-testing to execute the first formal correctness-testing phase for a SaaS application.

Goal:
- test only features that are already implemented and currently reachable
- focus on frontend-backend contract alignment, backend response correctness, permission correctness, scope isolation correctness, and basic functional correctness
- seed reasonable deterministic data before testing
- produce a test report and a bug report

Rules:
- do not do performance, stress, load, benchmark, or broad security testing
- do not auto-fix during the testing pass
- verify actual running behavior, not just code assumptions
- explicitly test both allowed and denied permission cases
- explicitly record setup assumptions and seed-data method
```

## 可复用背景信息

如果需要给智能体补充背景，可以使用：

```text
This is the first formal correctness-testing phase for a SaaS application. The goal is to verify already implemented functionality rather than expand scope. Prioritize real frontend-backend alignment, backend response correctness, permission enforcement, scope isolation, and reproducible findings. Treat hidden UI controls as insufficient evidence of access control. Confirm behavior through actual system execution.
```

## 完成定义

一次测试运行只有在以下条件全部满足时才算完成：

- 已确认当前可测范围
- 已准备并记录合理测试数据
- 已执行前后端正确性验证
- 已执行权限与作用域验证
- 已明确记录契约不一致问题
- 已产出测试报告
- 已产出 Bug 报告
- 未在测试阶段混入修复动作，除非用户另行要求
