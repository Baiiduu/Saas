# 需求落实度检查

## 概述

本技能用于把“需求文档”与“项目现实”逐项对照，判断需求是否真实落地。  
它不是普通代码审查，也不是功能测试，而是站在“需求到实现映射”的角度，检查需求完成性、实现偏差和验收可达性。

## 适用任务

在以下情况使用本技能：

- 用户要求检查一份需求文档是否已经落实到项目里
- 用户要求判断某一阶段需求完成了多少
- 用户要求对照 SRS、子需求文档、PRD、验收清单去查代码实现
- 用户要求产出需求追踪矩阵、完成度报告、缺口报告
- 用户准备测试、答辩、交付或评审，需要先确认“需求是否真的做到了”

不要在以下情况使用本技能：

- 用户只是要做代码风格 review
- 用户只是要做性能测试或安全测试
- 用户只是要头脑风暴新需求
- 用户只是要写新的需求文档，而不是检查落实度

## 角色定位

你在这一轮里不是开发者，也不是测试执行者。  
你是“需求落实审计者”。

你的职责是：

- 解析需求
- 建立需求与实现的映射关系
- 找到前端、后端、数据、权限、测试等各层证据
- 判断每条需求落实到什么程度
- 识别偏差、缺口、伪完成和不可验收项

除非用户明确要求，不要在这一阶段直接修复问题。

## 核心原则

### 1. 先拆需求，再查实现

不要直接全局搜索代码然后猜。  
先把需求拆成最小可核查单元，再逐项找证据。

最小可核查单元通常包括：

- 需求 ID
- 需求标题
- 关键动作
- 关键对象
- 关键约束
- 验收信号

### 2. 检查“真实落实”，不是检查“代码存在”

以下情况都不能直接认定为需求已完成：

- 有前端页面，但没有真实数据链路
- 有后端接口，但没有业务闭环
- 有数据表，但没有实际可用流程
- 有按钮或入口，但没有权限收口
- 有文档说明，但代码并未实现

### 3. 需求完成度必须分级

不要只给“完成 / 未完成”两档。  
默认使用以下分级：

- `已完整落实`
- `已部分落实`
- `仅原型/占位落实`
- `未落实`
- `无法判断，需补充证据`

### 4. 证据必须可追踪

每个判断都应尽量给出证据来源，例如：

- 前端页面
- 前端 service / hook
- 后端 controller
- 后端 service
- 数据模型 / schema
- 守卫 / 权限逻辑
- 测试用例
- 状态文档 / 变更说明

没有证据支撑的结论，只能写成“推测”或“待确认”，不能写成已确认结果。

## 标准工作流

### 第一步：识别输入材料

优先收集：

- 需求文档 / 规格说明书 / 子需求文档
- 项目代码库
- 接口文档
- 测试文档
- 状态文档
- 原型或页面入口

如果需求文档有明确 ID、阶段、优先级、验收条件，优先保留这些结构。

### 第二步：建立需求清单

把输入文档中的需求整理为清单。  
如果原文已有需求总表，优先直接提取。  
如果原文只是长篇散文，需要先拆成可核查条目。

至少保留：

- 需求 ID
- 需求描述
- 所属阶段
- 可验证对象

### 第三步：建立检查维度

默认从以下维度检查每条需求：

- 前端落实情况
- 后端落实情况
- 数据层落实情况
- 权限与作用域落实情况
- 测试或验收落实情况
- 用户可见闭环情况

不是每条需求都必须覆盖全部维度，但必须说明为什么。

### 第四步：逐项找实现证据

对每条需求，优先查：

- 入口是否存在
- 请求链路是否存在
- 响应链路是否闭环
- 数据是否真的持久化
- 权限是否真实收口
- 页面是否真正可用

如果只能找到局部实现，要明确说明“落实到了哪一层，缺了哪一层”。

### 第五步：判定完成度

按以下思路判断：

- 前后端闭环、数据闭环、权限闭环、用户闭环都成立：`已完整落实`
- 只完成了部分链路或少量场景：`已部分落实`
- 只有原型、Mock、占位页面或空接口：`仅原型/占位落实`
- 看不到实现证据：`未落实`
- 证据冲突或材料不足：`无法判断，需补充证据`

### 第六步：识别偏差和风险

除“有没有做”外，还要检查：

- 是否偏离需求原意
- 是否弱化实现
- 是否替代实现
- 是否存在前后端契约漂移
- 是否看起来完成但不可验收

### 第七步：输出结构化文档

默认输出两类结果：

1. 需求落实追踪报告
2. 需求缺口 / 偏差报告

## 推荐输出结构

### 输出 1：需求落实追踪报告

默认文件名：

- `REQUIREMENT_TRACE_REPORT.md`

建议至少包含：

- 检查范围
- 输入材料
- 检查方法
- 需求落实矩阵
- 已完整落实项
- 已部分落实项
- 原型/占位项
- 未落实项
- 无法判断项
- 总体结论

### 输出 2：需求缺口报告

默认文件名：

- `REQUIREMENT_GAP_REPORT.md`

建议至少包含：

- 缺口项清单
- 偏差项清单
- 风险项清单
- 影响范围
- 建议优先级

## 推荐矩阵格式

默认使用如下表格：

| 需求 ID | 需求描述 | 前端落实 | 后端落实 | 数据层落实 | 权限落实 | 测试落实 | 完成度 | 证据 | 备注 |
|---------|----------|----------|----------|------------|----------|----------|--------|------|------|

如有必要，也可拆成更细粒度版本：

| 需求 ID | 标题 | 阶段 | 页面入口 | 接口入口 | 业务逻辑 | 数据持久化 | 权限/作用域 | 测试/验收 | 完成度 | 说明 |
|---------|------|------|----------|----------|----------|------------|-------------|-----------|--------|------|

## 完成度判定规则

### 已完整落实

满足大多数以下条件：

- 需求入口存在
- 前后端链路闭环
- 数据持久化成立
- 权限与作用域基本成立
- 用户可实际完成完整操作
- 具备测试或可验收证据

### 已部分落实

常见情况：

- 只完成读链路，未完成写链路
- 只完成页面，未完成接口
- 只完成接口，未完成前端接入
- 只完成主流程，缺少边界或异常
- 权限未收口，导致暂不可正式验收

### 仅原型/占位落实

常见情况：

- 页面存在但数据是静态的
- 接口存在但返回模拟数据
- 结构有了，但没有真实业务规则
- 文档里写了，代码只是预留点

### 未落实

常见情况：

- 没有页面、没有接口、没有数据结构、没有逻辑证据

### 无法判断

常见情况：

- 需求描述本身过于模糊
- 项目中证据不足
- 只有局部材料，无法完成闭环确认

## 检查重点

### 1. 主路径闭环

优先判断用户能否真的完成需求描述的主要动作。

### 2. 权限与作用域

对企业级、多团队、多角色、多租户系统，必须重点检查：

- 是否只在正确作用域内生效
- 是否前端限制但后端未限制
- 是否有共享权限或资源级覆盖逻辑遗漏

### 3. 契约一致性

重点检查：

- 前端依赖的字段后端是否真的返回
- DTO 与实际请求是否一致
- 枚举、分页、错误格式是否一致

### 4. 验收可达性

某条需求即使看起来“有实现”，如果无法被正常测试和验收，也不能轻易判为完整落实。

## 不要这样做

- 不要只搜到文件名相似就判定完成
- 不要只看页面截图就判定完成
- 不要只看 controller 存在就判定完成
- 不要把“未来计划”写成“当前完成”
- 不要在没有证据时强行下结论

## 可直接复用的提示词

```text
Use $requirements-trace-audit to check whether a requirements document has been truly implemented in the project.

Goal:
- map each requirement to real implementation evidence
- determine whether each requirement is fully implemented, partially implemented, prototype-only, not implemented, or unclear
- identify implementation gaps, requirement drift, and non-acceptable pseudo-completion
- produce a structured requirement trace report and a requirement gap report

Rules:
- do not treat the existence of code, pages, or endpoints alone as completion
- check frontend, backend, data, permission, and test evidence when applicable
- provide traceable evidence for each conclusion
- focus on requirement-to-implementation mapping, not generic code review
```

## 完成定义

当且仅当满足以下条件时，本技能的一次使用才算完成：

- 已建立需求清单
- 已逐项检查实现证据
- 已按完成度分级
- 已指出偏差、缺口和风险
- 已形成需求落实矩阵
- 已输出追踪报告或缺口报告
